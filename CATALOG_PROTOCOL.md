## 1. עקרונות-על (חמשת חוקי הברזל)
1. **שם נקי ≠ שם נכון.** תקינות תחבירית לא מעידה על נכונות. רק השוואה למקור מוכיחה.
2. **לא ממציאים.** שם שלא ניתן לאמת מול PDF/source_truth — לא נוגעים בו (מעבר לניקוי בטוח).
3. **המק"ט (`sku`) קדוש.** מזהה ייחודי, לא משוכפל, לא משתנה.
4. **כל שינוי עובר רגרסיה.** `audit` + רגרסיית "קטלוג" באפליקציה ירוקה לפני commit.
5. **אותו פרוטוקול בכל קנה מידה.** 800 או 80,000 — אותם כלים, אותם שלבים.

### `LipskeyCatalogProduct` — מילון שדות
| שדה | סוג | חובה | תיאור |
|-----|-----|------|-------|
| `sku` | String | ✅ | מק"ט ייחודי (גם אלפאנומרי: `77Z2081C`) |
| `nameHe` | String | ✅ | שם עברי (single-quote; אם יש `'` → double-quote) |
| `nameEn` | String | ✅ | שם אנגלי |
| `categoryHe`/`categoryEn` | String | ✅ | קטגוריה (מחרוזת עקבית!) |
| `categoryEmoji` | String | ✅ | אימוג'י קטגוריה |
| `page` | int | ✅ | עמוד במקור (ל-`specImageAsset`) |
| `imageFile` | String? | — | `<sku>.jpg` או null |
| `brand` | String | — | `'AQUATEC'` / `'ליפסקי'` (ברירת מחדל) |
| `color`,`qtyPack`,`qtyPallet`,`dims` | — | — | מפרט: צבע, אריזה, משטח, מידות |

נגזרות (getters): `imageAsset` · `specImageAsset` (`assets/lipskey/pages/page_NN.jpg`)
· **`typeEmoji`** (אייקון לפי סוג כשאין תמונה).

### `CatalogNode` (עץ)
`id` (היררכי בנקודות: `drainage.pipes.pvc`) · `title` · `emoji` · `children`
· `brandIds` · **`lipskeyCategory`** (→ מסנן מוצרים) · **`smartKey`** (→ SmartProduct).
`isLeaf` = אין children.

### `SmartProduct` / `Brand`
SmartProduct: `key,name,emoji,cat,brands[],acc[],stages[]` — כרטיס בחירת מותג+אביזרים.
Brand: `id,name,emoji,color,tagline,productCount`.

## 3. המקורות והמותגים
- ליפסקי = **מפיץ**, לא יצרן יחיד. שני מותגים בקטלוג:
  - **ליפסקי** (~255): ניקוז · מחסומים · סיפונים · אסלות · מיכלי הדחה · צינורות · תבריג · אטמים · מאספים · מסעפים · ברכיים.
  - **AQUATEC** (601): ברזים · מקלחות · מזלפים · נחושת · HDPE · NTM · חבקים · מחלקים · גינה · אביזרי חדר רחצה.
- המקור = **4 PDFs**. כרגע בידינו 1 (קטלוג ניקוז ליפסקי, 30 עמ').
- ⚠️ **מספור עמודי ה-PDF שונה משדה `page` בנתונים** (נשאב מ-PDF אחר/מורחב). אל תסמוך על התאמה — חפש לפי SKU.

## 4. זרימת המשתמש (צלילה → מוצר)
```
טאב קטלוג → "קטגוריות" → קטגוריה ראשית → תת → עלה
   leaf.lipskeyCategory יש מוצרים? ──כן──→ LipskeyProductsScreen (רשימה, כרטיס חיצוני)
                                  └─לא──→ openSmartProductSheet (fallback grohe/hamat)
   לחיצה על מוצר → showLipskeyProductSheet (כרטיס מלא)
```
**כרטיס חיצוני** (`_ProductRow`): תמונה(✓-בסל / זום), שם 2-שורות, מילים לחיצות (חיפוש),
מק"ט(העתקה), "+"(בורר כמות בודד/ארגז/משטח, מוסיף לסל בלי לפתוח כרטיס), ⓘ פרטים.
**כרטיס מלא** (`LipskeyProductSheet`): X · תמונה↔מפרט (היפוך+זום) · בחר גרסה ·
אביזרים · שלבי-התקנה · כמות+יחידות · **הוסף לסל** · **חיבורים תואמים** · נלווים.

## 5. מנוע התאימות — "מה מתחבר למה" (לב המוצר)
- כל מוצר נושא **מידות DN** בשם: `DN50`, `50/40`, `130/50`, `3/4"`.
- `_sizeSet(name)`: מחלץ מידות **בהקשר ממדי בלבד** (DN-קידומת, יחס `/`, אינץ' `"`).
  **מתעלם** ממספרים בודדים (כמות אריזה "75", אורך "200 ס\"מ").
- `_connectionSizes`: קצוות אטומיים. מצמד `75/50` = שני צדדים {75,50}; `DN50` = צד אחד.
- בכרטיס: סקשן "חיבורים תואמים" **לכל צד** → כל החלקים מקטגוריות אחרות
  שחולקים את המידה (חיתוך קבוצות). דוגמה: מעבר 75/50 → צד DN75 (ברכי 75) + צד DN50 (אטמי 50).
- **לכן `NOSIZE` משמעותי**: מוצר בלי DN בשם לא משתתף בתאימות → להשלים מהמקור.

## 6. תמונות וזיהוי (שרשרת fallback)
```
imageFile != null  →  assets/lipskey/products/<sku>.jpg   (תמונת מוצר)
        ↓ אין/נכשל
typeEmoji          →  אייקון לפי סוג: ניפל🔩 מצמד🔌 צינור📏 ברך↩️ מסעף🔱
                      מכסה⬜ פקק⚫ מושב⭕ מחסום🕳️ מזלף🚿 חבק🪝 מחלק🔀 ...
        ↓ דרך כפתור "פרטים/מפרט"
specImageAsset     →  crop עמוד ה-PDF (page_NN.jpg) + זום מלא
```
**כלל**: לעולם לא אימוג'י קטגוריה אחיד למוצר ללא תמונה (כל הנחושת=🟫 → אי-אפשר להבחין).
`typeEmoji` נגזר ממילת-הסוג בשם → כל מוצר ניתן לזיהוי גם בלי צילום.

## 10. `scripts/source_truth.json` (יופק ע"י הסוכן עם ה-PDFs)
מערך, רשומה לכל מוצר:
```json

```
כללים: מק"ט ייחודי · שם קריא ונכון מהעמוד (בלי זבל אריזה, אינץ'=`"`) · `sizes` מערך
מידות חיבור · קטגוריה עקבית. כשמגיע → `truthcheck` חושף ומתקן הכל ב-100%.

## 11. תיקוני נתונים בטוחים (Python, על `nameHe` בלבד)
- נרמול: `״ “ ”` → `"`. להשאיר `׳` (ג׳קוזי לגיטימי).
- הסרת מקטעי אריזה: `"NN כמות באריזה/במשטח"`.
- פיצול `|` — לאחד רק חלקים תיאוריים.
- רווח בגבול ספרה↔אות (`1.5מחסום`→`1.5 מחסום`) **בלי לשנות סדר** (לא לשבור מילה!).

## 12. מלכודות (gotchas) — נלמדו בדם
1. **גרש Dart**: `'` בודד במחרוזת single-quote שובר קומפילציה. שם עם גרש → `"..."`.
2. **RTL+מספרים**: חילוץ PDF מקושקש ("1.5מחסום ״"). חובה verify ויזואלי לשם מדויק.
3. **מספור עמודים**: `page` בנתונים ≠ עמוד ב-PDF הנוכחי. חפש לפי SKU.
4. **`brand` ברירת מחדל** = 'ליפסקי'. AQUATEC מסומן מפורשות.
5. **שם כפול ≠ שגיאה**: וריאציות אמיתיות (מחבר כפול DN75 ×2) — מק"ט מבדיל.
6. **מספר בודד אינו מידה**: "75 כמות באריזה" / "200 ס\"מ" — לא DN.
7. **double-quote בשם** (ג'נבה) נקרא "ריק" ע"י regex של single-quote → false-positive.

## 13. הוספת קטלוג/מותג חדש (end-to-end)
1. הוסף PDF → הרץ extractor (`scripts/extract_*.py`) → רשומות `LipskeyCatalogProduct`.
2. `Brand` חדש ב-`brands.dart` (`brand:` נכון לכל מוצר).
3. `categoryHe` חדשות → מַפֵּה לעלים ב-`catalog_tree.dart` (`lipskeyCategory`+`smartKey`).
4. `audit` → תקן תחביר. `crosscheck <pdf>` / `truthcheck` → תקן סמנטיקה.
5. `flutter analyze && flutter build web --release` → רגרסיה ירוקה → commit.
